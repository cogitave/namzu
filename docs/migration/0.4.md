# Migrating to `@namzu/sdk@0.4.0`

`0.4.0` is a minor release under pre-1.0 semver that is explicitly allowed
to break. It prunes `@namzu/sdk`'s runtime dependencies from 10 to 0 by
extracting the OpenTelemetry pipeline into a new package and moving two
packages to peer deps. No feature changes. No provider-package changes.

Consumers installing the SDK for the first time post-`0.4.0` will pull
in far less â€” `@opentelemetry/api` + `zod` + `zod-to-json-schema` only,
if they use those surfaces.

This page is the contract. Every removed symbol has a replacement
listed below; every supported upgrade path is stated explicitly.

## Section A â€” If you use telemetry

Before (`0.3.x`):

```ts
import { initTelemetry, createPlatformMetrics } from '@namzu/sdk'

const telemetry = initTelemetry({ serviceName: 'app', exporterType: 'otlp', /* ... */ })
await telemetry.start()

const metrics = createPlatformMetrics()
```

After (`0.4.0`):

```
pnpm add @namzu/telemetry @opentelemetry/api
```

```ts
import { registerTelemetry, createPlatformMetrics } from '@namzu/telemetry'

const telemetry = await registerTelemetry({
  serviceName: 'app',
  exporterType: 'otlp',
  /* ... */
})

const metrics = createPlatformMetrics()
```

Two breaking changes in this path:

1. **`initTelemetry` â†’ `registerTelemetry`.** The function was renamed
   and made asynchronous. It returns `Promise<TelemetryProvider>` and
   must be `await`ed. You no longer call `provider.start()` separately â€”
   `registerTelemetry` does it internally before resolving.
2. **Import source.** All telemetry symbols move from `@namzu/sdk` to
   `@namzu/telemetry`.

See [Section E](#section-e--full-symbol-mapping) for the full list.

## Section B â€” If you write tools with Zod

`zod` and `zod-to-json-schema` are now `peerDependencies` of
`@namzu/sdk`. You almost certainly already have them installed
transitively, but declare them explicitly to be safe:

```
pnpm add zod zod-to-json-schema
```

No code changes. `defineTool({ params: z.object({ ... }) })` continues
to work exactly as before. Peer range: `zod@^3.23.0`,
`zod-to-json-schema@^3.23.0`. Zod 4 support is deferred to a follow-up
matrix test; do not upgrade to `zod@^4` yet.

## Section C â€” If you don't use telemetry

`@opentelemetry/api` is a new peer of `@namzu/sdk`. The SDK's internal
runtime (run spans, iteration spans, tool-execution spans) still calls
into `@opentelemetry/api`, but without a registered provider those spans
go to the OTEL no-op tracer and are silently discarded â€” same behavior
as `@namzu/sdk@0.3.x`, just via a peer dep now.

```
pnpm add @opentelemetry/api
```

No code changes.

## Section D â€” "Spans seem to disappear"

If spans from your code or from the SDK runtime are not reaching your
observability backend, check in order:

1. **Did you `await registerTelemetry(...)`?** The function is
   asynchronous. Without the `await`, startup races your first span
   and may silently drop early traces.
2. **Did you `registerTelemetry` before `createPlatformMetrics`?**
   `createPlatformMetrics` binds counters at construction time. If you
   constructed it first, every `.record()` is going to the no-op meter
   forever â€” registering a provider later will *not* rewire existing
   counters.
3. **Is `@opentelemetry/api` deduplicated in your install tree?** Peer
   dependencies must resolve to a single physical module. Run
   `pnpm why @opentelemetry/api` or `npm ls @opentelemetry/api` â€” more
   than one version means split tracer globals and silently lost spans.
   The CI fixture `.github/scripts/verify-consumer-install.sh` asserts
   this invariant on every release.
4. **Are you mixing `@namzu/sdk@0.3.x` and `@namzu/sdk@0.4.0` in the
   same dependency tree?** Pin one version across your workspace.

## Section E â€” Full symbol mapping

Every symbol removed from the `@namzu/sdk` root barrel has a replacement.
If a symbol below is not in your migration, you don't need to change
anything for it.

| Removed from `@namzu/sdk` | Import from `@namzu/telemetry` |
| --- | --- |
| `TelemetryProvider` (class) | `TelemetryProvider` |
| `initTelemetry(config)` (sync, returned provider) | `registerTelemetry(config)` (async, `Promise<provider>`) |
| `getTelemetry()` | `getTelemetry()` |
| `getTracer()` | `getTracer()` |
| `getMeter()` | `getMeter()` |
| `createPlatformMetrics()` | `createPlatformMetrics()` |
| `PlatformMetrics` (type) | `PlatformMetrics` |
| `TelemetryConfig` (type) | `TelemetryConfig` |
| `ExporterType` (type) | `ExporterType` |
| `GENAI`, `NAMZU` (attribute constants) | `import { GENAI, NAMZU } from '@namzu/telemetry/attributes'` |
| `agentRunSpanName`, `agentIterationSpanName`, `chatSpanName`, `toolSpanName` | same names, from `@namzu/telemetry/attributes` |

The only signature-changing migration is `initTelemetry` â†’
`registerTelemetry`:

- Old: `const t = initTelemetry({...}); await t.start(); /* use t */`
- New: `const t = await registerTelemetry({...}); /* use t */`

Every other symbol keeps its name.

## What did NOT change

- `defineTool` signature and semantics.
- Provider-registration APIs (`registerOpenAI`, `registerAnthropic`, etc.).
- Run/iteration/tool event shapes.
- Session hierarchy types.
- Any SDK surface not in the table above.

## What's coming later (not in `0.4.0`)

- **`withTelemetry(provider)` wrapper.** A future session adds an
  opt-in HOF that wraps any `LLMProvider` to emit per-call GenAI-semconv
  spans. Not shipped in `0.4.0`; the `@namzu/telemetry` package exists
  but the wrapper is explicitly deferred until the provider
  `StreamChunk.usage` contract is ratified.
- **Zod 4 support.** Deferred to a matrix-test session.
- **SDK 1.0 cut.** Will land after `ses_011-sdk-public-surface` purges
  deprecated aliases like `mapSessionToA2AEvent` and the transitional
  provider-factory types.
