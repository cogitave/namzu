## Status

Proposed.

## Context

`@namzu/sdk@0.1.x` governs tool invocation through a scalar `PermissionMode: 'plan' | 'auto'` evaluated synchronously at `packages/sdk/src/registry/tool/execute.ts:230-248` against `ToolDefinition.isReadOnly(rawInput)`. Secondary enforcement lives in `VerificationGate` at `packages/sdk/src/verification/gate.ts`, also synchronous, rule-matching on tool name and input shape. Five structural problems make this model unable to support the kernel's forthcoming collaboration and memory surfaces:

1. **Two values cannot express three realistic defaults.** `always_allow`, `always_ask`, `never` are the minimum; conditional rules are the next rung. A scalar string cannot encode this.
2. **Sync evaluation excludes non-trivial rules.** Regex-on-path checks fit; tenant-scoped allowlist lookups, remote policy servers, and per-tenant HITL escalation do not.
3. **Unguarded predicates crash runs.** `tool.isReadOnly(rawInput)` throws on malformed input and there is no wrapper.
4. **`defineTool({ destructive: fn })` encourages impure predicates** inside a value that should be a static capability tag.
5. **Tools are the only governed surface.** Handoff (Task 0a), merge (Task 0b), session archival, runtime-mode flips, and memory (Task 5) have no policy surface — each ships its own ad-hoc gate or relies on the platform above the SDK.

Task 0b (`collaboration-primitives.md`) explicitly depends on a shared policy engine for `merge.*` gating; Task 5 (memory) depends on the same for `memory.*`. Neither can code against today's surface without inventing a bridge — a Convention #0 breach.

## Decision

Introduce a declarative, typed, inheritance-resolved, deny-by-default policy engine at `packages/sdk/src/permission/`. Policy values are the discriminated union `'always_allow' | 'always_ask' | 'never' | { type: 'conditional', rule: PredicateRule | ExternalRule }`. A central `PolicyEvaluator` resolves effective policy bottom-up through tiers `run > session > tool > toolset > kernel`, with `'never'` absolute at any tier (safety invariant — session override cannot escape toolset `never`). All evaluation is asynchronous, timeout-bounded, and error-unified: sync throw, async throw, Promise rejection, timeout, cycle, or depth overflow all collapse to `deny` with a structured `DenyReasonCode`. A pure `previewPolicy()` API returns a **structured `PreviewDecision`** (`allow | ask | deny | { unknown, reason: 'impure' | 'external' | 'timeout' | 'async_required' }`), never invokes external backends, and feeds a stable `permissionPreviewFingerprint()` cache key. The action catalog spans six namespaces (`tool`, `handoff`, `merge`, `session`, `mode`, `memory`) each with a kernel default; Task 0b and Task 5 register defaults at module init via `ToolPermissionRegistry.registerAction(namespace, policy)` (late-bindable; idempotent on match, rejects on mismatch). Tool-level risk surfaces become TWO orthogonal signals: static `destructive: boolean` (declared intent) plus a pure `riskFn(projectedInput) → 'none' | 'low' | 'medium' | 'high'`; kernel default becomes `destructive OR riskFn ∈ {medium, high} → always_ask`. A first-class `inputProjection(input) → unknown` on `ToolDefinition` populates `PermissionContext.projectedInput` so predicates decide on input shape WITHOUT seeing raw input. Plugin capabilities (`readsPermissionInput`, `rewritesPermissionInput`, `evaluatePermission`) require an explicit `TenantConfig.allowedPluginCapabilities` allow-list at load; violations emit `plugin_load_denied` (degraded-continue) or abort boot under `strictPluginCapabilities` (Gap M). `DenyReasonCode` splits `plugin_capability_denied_at_load` (load-time) from `plugin_hook_threw_at_runtime` (runtime hook exception) (WEAK #2). A new `pre_policy_eval` pipeline stage lets capability-gated plugins normalize input BEFORE the evaluator runs. Runtime modes (`askPermissions | acceptEdits | bypassPermissions`) layer on top of policy — `mode.change` is itself a governed action with a five-step resolver fallback chain terminating at `TenantConfig.modeChangeResolverId: ActorRef` (required at tenant provisioning; no magic "tenant-owner principal"). Mode propagates via a fixed matrix (project→session→run) with snapshot-at-spawn semantics (no retroactive parent propagation). Audit events gain `resolutionTier: PolicyTier`, `schemaVersion: 2`, and new variants: `mode_transition` (single event with `outcome: 'applied' | 'denied' | 'cascade_applied'` discriminant — v3 Gap J), `namespace_registered`, `action_registered`, `external_backend_trip`, `external_backend_override`, `permission_escalation_unresolvable`, `permission_escalation_timeout`, `permission_escalation_cascade_applied`, `permission_resolved`, `projection_fail`, `plugin_load_denied`. All carry only a tenant-salted `inputHash`, never raw input; redaction invariants are enforced at `store/run/disk.ts:36-44` and `bridge/sse/mapper.ts:45-51` before emit. Cross-tenant audit correlation is an explicit non-goal. Cycle detection via `InvocationState.permissionStack` and `maxPolicyDepth: 8` prevent recursion DoS; stack is ephemeral (never checkpoint-serialized). External rules ship a default circuit breaker (`3 failures / 30s coolDown / fallbackPolicy: 'deny'`) with admin-override auto-expiry. Effective-policy memoization at `(actionRef, tenantId, staticPolicyTreeVersion)` keeps hot paths O(1); any registration mutation bumps the version. HITL uses typed `permission_request` / `permission_resolve` envelopes under `types/hitl/`; `HITLConfig.onTimeout` is the discriminated union `'deny' | 'pause_run_until_resolved' | { kind: 'cascade', fallback, afterMs }`; `pause_run_until_resolved` transitions the Run to `awaiting_hitl_resolution`. `VerificationGate` becomes a **fail-fast** deprecated facade over `PolicyEvaluator` that throws `VerificationGateIncompatible` on any async/external rule at construction. All principal fields unify on `ActorRef` (imported from session-hierarchy.md §4.3); `PrincipalRef` is retired. Full design, resolution matrix, pipeline diagram, module tree, and verification checklist are in the pattern doc.

## Consequences

### Breaking impacts (top 3 — ordered by blast radius)

1. **Tool-execute path is now async end-to-end, with a new `pre_policy_eval` stage.** This is the **largest blast radius** — every call site that today performs a sync policy check migrates. `registry/tool/execute.ts:230-248`, `runtime/query/iteration/phases/tool-review.ts:43-50`, and `runtime/query/executor.ts:129-158` all switch from sync checks to `await evaluator.evaluate(...)`. Every consumer of `VerificationGate.verify` along with downstream code that synchronously unwrapped its result must now await. Every sync thrown-predicate crash converts to a structured `deny`. Pipeline ordering is load-bearing: JSON parse → `pre_policy_eval` (capability-gated raw-input normalization) → `tool.validate` (ONLY mutating/annotating stage; I/O allowed; `ValidatedInput` frozen on return) → `inputProjection` → `resolveStaticPolicy` → `riskFn` → `applyRiskOverride` → `evaluator.evaluate` (predicates/external/HITL) → on allow/ask `pre_tool_use` (observe/veto) → execute → `post_tool_use`. On `deny`, `pre_tool_use` is skipped entirely, so plugins that relied on observing denied invocations must subscribe to `permission_denied` audit events instead. Consumers enumerated in pattern doc §16 must update atomically in the same PR; `_exhaustive: never` guards enforce at compile time.
2. **`VerificationGate` is deprecated as a fail-fast facade.** The class remains exported from `packages/sdk/src/verification/index.ts` for 0.2.x as a strictly fail-fast facade over `PolicyEvaluator`, emitting `MigrationWarning` once per process on construction. External rules and async predicates cannot be faithfully downgraded — attempting to construct a `VerificationGate` over any async/external rule throws `VerificationGateIncompatible { unsupportedRules }` at construction time (no silent async-to-sync coercion, no hidden promise waits). Only fully-legacy sync-equivalent policies admit construction during the migration window. Removed in 0.3.0. `packages/sdk/src/types/verification/` rule types migrate to `packages/sdk/src/permission/types/policy.ts`.
3. **`ToolDefinition.destructive: (input) => boolean` is removed; static `boolean` only.** Every tool definition today that passes a function for `destructive` (see `src/tools/defineTool.ts:17-18`) fails to compile in 0.2.0. Migration has two paths: (a) declare `inputProjection` + `riskFn` so the kernel synthesizes the default automatically (preferred for input-shape-conditional risk); (b) register a `{ type: 'conditional', rule: { type: 'predicate', fn, pure: true } }` at tool tier where the predicate reads `ctx.projectedInput` (never raw input). For the common path-prefix case the pragmatic migration is `destructive: true` + tier policy `'always_ask'` — the policy engine does not second-guess "is this input safe", that's a HITL concern. Migration-warning once per process while the deprecation window is open (removed 0.3.0).

### New invariants introduced (v2 + v3)

- **Input projection is the contract for all tools with conditional policies.** `ToolDefinition.inputProjection(input) → unknown` runs pure + deterministic at context-build; its output populates `PermissionContext.projectedInput`. Predicates and `riskFn` read only this — never raw input. The raw-input capability path exists only as a last-resort plugin escape valve gated by `TenantConfig.allowedPluginCapabilities`.
- **Structured `PreviewDecision` with discriminated `reason`.** Replaces flat `'unknown'`. `{ unknown, reason: 'external' }` preserves cross-turn cache-hit rate (fingerprint stable on backend identity); `{ unknown, reason: 'timeout' }` intentionally bypasses caching.
- **Ephemeral `permissionStack`.** Cycle-detection state is never serialized to checkpoints; restored stacks are empty. Checkpoint boundaries never fall inside evaluation.
- **`pre_policy_eval` hook + capability-gated plugin mutation.** Input normalization happens in its own pipeline stage, gated by the `rewritesPermissionInput` capability AND tenant allow-list. `pre_tool_use` becomes strictly observe/veto — input-mutation capability lives at `pre_policy_eval` only.
- **Mode propagation matrix (project → session → run hierarchy).** Propagation is explicit: project → session → sub-session → run. Snapshot-at-spawn; later parent changes do NOT retroactively propagate. Run-scope is terminal.
- **Mode-transition audit event contract.** `mode_transition` emitted from `modeStore.set()` unconditionally — success OR failure (failure carries `{ denied: true, denialReason }`). Operator-supplied `reason` passes through the same redaction invariants as other audit fields.
- **Plugin capability allow-list as tenant config.** `TenantConfig.allowedPluginCapabilities: ReadonlySet<PluginCapability>` + `allowedThirdPartyNamespaces: ReadonlySet<string>` gate plugin load; violations throw `PluginCapabilityDenied`.
- **Memoized effective-policy resolution.** Keyed by `(actionRef, tenantId, staticPolicyTreeVersion)`. Version bumps on any `registerAction`, `registerNamespace`, `registerSpecific`, or `unregister`. Runtime-conditional decisions still evaluate per-call.
- **`never` is absolute.** Scanned across every tier before most-specific resolution runs; no override escapes it.
- **All evaluation is async + timeout-bounded + error-unified.** Any failure path produces a structured `PermissionDecision.deny` with a typed `DenyReasonCode`; no raw throw escapes.
- **`PermissionContext` carries `actor: ActorRef` + `tenantId` + `inputHash` + `projectedInput` + `invocationState`**. No `scope` field — origin tier is carried on `PermissionDecision.resolutionTier` / audit events.
- **Cycle detection is first-class.** `InvocationState.permissionStack` + `maxPolicyDepth: 8` (configurable).
- **`mode.change` is a governed action with a required `TenantConfig.modeChangeResolverId`.** Five-step resolver fallback chain (`HITLConfig.resolverActor` → `parentSession.currentActor` → ancestry walk → `tenantConfig.modeChangeResolverId` → deny final with `permission_escalation_unresolvable`).
- **External rule circuit breaker + tenant-admin override.** Default `{ failureThreshold: 3, coolDownMs: 30000, fallbackPolicy: 'deny' }`. `TenantConfig.externalPolicyOverride` supports temporary auto-expiring override; emits `external_backend_trip` / `external_backend_override` events.
- **HITL escalation is typed.** `types/hitl/index.ts` owns `permission_request` / `permission_resolve`; `onTimeout` discriminated union supports `'deny'` (default), `'pause_run_until_resolved'` (parks run in `awaiting_hitl_resolution`), or `{ kind: 'cascade', fallback, afterMs }`.
- **Action namespaces are exhaustive.** Adding `memory` in Task 5 requires every §16 consumer to switch a new case in the same PR — the compiler enforces; the convention documents it. `registerNamespace` supports third-party extensions, gated by tenant-admin allow-list.
- **Cross-tenant audit correlation is a non-goal.** `inputHash` is tenant-salted; events are tenant-scoped. Any cross-tenant forensics requires a higher-layer aggregator; the kernel offers no mechanism.
- **`ActorRef` unified.** All principal fields across this doc AND cross-linked events use `ActorRef` from session-hierarchy.md §4.3. `parentActor?` chains subagent attribution on `memory.*` / tool events.

**v3 additions (from round-2 adversarial review):**

- **Durable `PendingHITLStore` (Gap G).** Mandatory primitive for `pause_run_until_resolved` and cascade timeouts. Disk-backed reference implementation (write-tmp-rename per #8); pluggable. Boot-time rehydration via `RunSupervisor.listByStatus('open')` re-arms wakeups; external `store.resolve` drives `Run.resume`. Without this store, `pause_run_until_resolved` is unimplementable.
- **`tool.validate` I/O phase (TRAP #1).** The ONLY pipeline phase permitted to perform I/O. `validate(input)` runs AFTER `pre_policy_eval` (so `validate` sees already-normalized input) and is the LAST mutating stage — kernel deep-freezes the returned `ValidatedInput` for all downstream stages. Returns `ValidatedInput { input, __namzu? }` (with precomputed snapshots under `__namzu.precomputed`) or denies with `validation_failed`. `inputProjection` and `riskFn` run on the frozen `ValidatedInput` and remain strictly pure. Root-cause fix for the purity discipline per #0 — impure pre-work has a dedicated phase.
- **Two-phase resolution (Gap B).** Phase 1 `resolveStaticPolicy(actionRef, tenantId, version)` is memoized; Phase 2 `applyRiskOverride(staticPolicy, projectedInput, toolDef)` is per-call (input-dependent). Absolute-deny wins: static `never` cannot be overridden by a risk upgrade. Single-flight invalidation per key prevents thundering-herd on version bumps; version bumps mark stale but do not purge.
- **`mode_transition` outcome discriminant (Gap J).** `changeMode()` emits exactly ONE event per attempt (N for cascade with shared `cascadeGroupId`) with `outcome: 'applied' | 'denied' | 'cascade_applied'`. Denial path is PURE AUDIT — `modeStore` does NOT mutate on denial. Consumer switches MUST handle all three variants (#6).
- **Admin cascade override (Gap H).** `ModeMutation.cascade: 'none' | 'live_descendants'` allows admin-only propagation to live descendants, governed by a distinct action `mode.change.cascade` (kernel default `always_ask`). Cascade does NOT apply to completed/archived sessions; idempotent. Cascade is the only exception to snapshot-at-spawn.
- **Namespace ownership binding (TRAP #3).** `allowedThirdPartyNamespaces` carries `{ namespace, ownerPluginId }` pairs; `registerNamespace` rejects with `NamespaceOwnershipConflict` unless the calling plugin's id matches. Prevents one admitted plugin from squatting another's namespace.
- **Pipeline reorder: `pre_policy_eval` now runs before `validate`; `validate` output is frozen for downstream stages.** Raw-input mutation lives in `pre_policy_eval` only; `validate` is the LAST mutating stage and kernel deep-freezes the `ValidatedInput` so projection / riskFn / predicates cannot observe mid-pipeline drift.
- **`ValidatedInput` reserved namespace: `__namzu` key is pipeline-only; tool-author fields must live under `input`.** Tools writing to `__namzu` at `validate` return are rejected with `ReservedNamespaceViolation` / `reserved_namespace_violation`. Precomputed snapshots go under `__namzu.precomputed`; downstream stages read snapshots by key, never from arbitrary `input` fields.
- **Cascade is audit-only correlated; no rollback primitive — administrators reconcile partial cascades manually.** `cascadeGroupId` correlates per descendant `mode_transition` events; a single `cascade_completed` summary event per walk reports `totalAttempted / applied / denied / deniedDescendants`. `mode_transition { outcome: 'cascade_applied' }` fires at the next-tool-boundary (observable moment), not at cascade invocation.
- **`SessionStore.deriveStatus` + restart-recovery repair.** Session status is derived from runs + open HITL requests per the session-hierarchy v3 §5.1 fan-in rule. On kernel boot, `RunSupervisor` reconciles stored `Session.status` against `deriveStatus` and emits `session_status_repaired` per repair, ensuring coherence across restart.
- **Plugin hook runtime throws carry a `pluginContext` discriminant on `permission_denied`.** Present iff `reasonCode === 'plugin_hook_threw_at_runtime'`, carries `{ pluginId, hook, errorClass }`. Other denial paths omit the field.

### Non-breaking impacts

- Existing `ToolDefinition` shape (name, description, inputSchema, category, permissions, execute) is unchanged beyond the `destructive: fn` narrowing and the NEW optional `inputProjection` / `riskFn` fields.
- Plugin hook contract (`pre_tool_use`, `post_tool_use`) is unchanged; a new `pre_policy_eval` hook is additive and opt-in via capability declaration.
- Run-internal persistence (`run.json`, `messages.json`, `transcript.jsonl`) is unchanged; audit events land on the kernel bus through existing channels with new discriminated variants handled by `§16`'s consumer set.
